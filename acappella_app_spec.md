# アカペラ練習サポートWebアプリ
## 要件定義 / 開発方針書 v2.1

---

## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| 目的 | アカペラグループの練習効率化（録音解析・音取り・テンポ確認を一元化）および将来的なデファクトスタンダードツールへの成長 |
| ターゲット | **フェーズ1〜2：** 友人グループ（クローズドな配布、限定利用）／ **フェーズ3以降：** アカペラ活動者全般（学生・社会人）への公開展開 |
| 提供形態 | ブラウザで動作するWebアプリ（PWA対応）／将来的にGoogle Play配信・iOS PWA配布を視野に入れる |
| 利用端末 | **スマホファースト**（全機能がスマホから操作可能であること） |
| ネットワーク | **オンライン前提**（オフライン対応は不要。利用者にも周知） |
| 練習頻度 | 月2〜3回、1回あたり最大3時間の音源 |
| 収益モデル | 広告収益（アカペラ関連イベント・楽部の広告掲載）を将来的に導入 |

## 2. 機能要件

### 2.1 録音解析機能（AIメイン機能）

#### 処理フロー（二段階方式）

```
音源アップロード
　↓
① 文字起こし（Gemini API）
　↓
② 構造化解析（Gemini API）
　　- パート別自動タグ付け
　　- カテゴリ分類
　　- 重要度判定
　　- タイムスタンプ紐付け
　↓
カード型リスト表示
　↓
JSONエクスポート
```

**二段階にする理由：**
- APIコストの最適化（音声を直接投げ続けるより安価）
- 文字起こし結果を画面表示でき、解析精度の確認が可能
- 同じ文字起こし結果でプロンプトのみ変えて再解析できる（開発・調整コスト削減）

#### 解析出力の構造

```json
{
  "session_name": "2025-07-12 練習",
  "recorded_at": "2025-07-12",
  "transcript": "（文字起こし全文）",
  "cards": [
    {
      "id": "001",
      "timestamp_sec": 342,
      "part": ["ベース", "バリトン"],
      "category": "ピッチ",
      "importance": "high",
      "text": "サビ入りの音程が毎回フラットになっている"
    }
  ]
}
```

#### 解析カードの属性

| 属性 | 内容 |
|---|---|
| timestamp_sec | 発言の秒数（音源シーク用） |
| part | 対象パート（ソプラノ / リード / バリトン / ベース / 全体 等） |
| category | ピッチ / リズム / ダイナミクス / 歌詞・発音 / その他 |
| importance | high / normal（「重要」「絶対」等のキーワードまたはAI推定） |
| text | 指示内容のテキスト |

#### フィルタリング

- パート別・カテゴリ別のチップ（ボタン）で絞り込み
- 複数選択時は**AND**（絞り込み）
- 1タップで全フィルター解除

---

### 2.2 過去解析の再現機能

#### 再現フロー

```
初回：音源 → AI解析 → カード表示 → JSONエクスポート（手動保存）
再現：JSON + 音源を再アップロード → カード表示（API呼び出しなし）→ タップでシーク
```

#### UI

```
[ 解析JSONを選択 ]　[ 音源ファイルを選択 ]　→ 読み込む
```

#### 突合ルール

- ファイル名による突合は**行わない**
- ユーザーが正しい組み合わせで選択する前提
- 誤った組み合わせをした場合は「無関係な場所の音が再生される」だけであり、データ破損等は発生しない
- **JSONエクスポート時のデフォルトファイル名を `[セッション名]_[日付]_解析結果.json` 形式にすることで、ユーザー側での管理ミスを軽減する**

### 2.3 ピッチパイプ機能

| 項目 | 仕様 |
|---|---|
| 操作 | 12音ボタン。タップで発音ON/OFFトグル（長押しは使用しない） |
| 複数同時発音 | 対応済み（和音・複数パートの入り音同時再生が可能） |
| 基準周波数 | 420〜460Hz可変、スライダー＋微調整ボタン |
| 起動時デフォルト | 設定画面で指定した値（未設定時は440Hz） |
| 音色 | サイン波 / 三角波 / 矩形波 の切り替え。起動時デフォルトは設定画面で指定 |

> **注意：** 長押しはブラウザのコンテキストメニューと競合するため採用しない。  
> `touchstart` / `touchend` + `preventDefault()` ではなくトグル方式を採用する。

### 2.5 曲ライブラリ機能 ※新規

ライブ・本番前の音出し確認に特化した機能。練習中タブとは独立した別画面として提供する。

#### 画面階層

```
曲ライブラリ画面
　└ グループ一覧
　　　└ 曲一覧
　　　　　└ 曲詳細・再生画面
　　　　　　　└ 編集画面
```

#### グループ

- グループ単位で曲を管理（掛け持ちメンバーを想定した複数グループ対応）
- グループの新規作成・編集・削除が可能

#### 曲詳細・再生画面

- 登録済みのキー音（複数可）・BPM・基準周波数で即座に再生／停止
- 画面を閉じるまで再生状態を保持（何度でも再生し直せる）
- 「編集」ボタンで編集画面へ遷移

#### 編集画面（登録時も同じUIを使用）

- 小型ピッチパイプUI（音名ON/OFFで複数音選択可能）
- BPM調整（数値入力 ＋ ±ボタン）
- 基準周波数調整（440Hz固定ではなく曲ごとに保存可能）
- 試し鳴らし（編集画面内でピッチ・BPMを確認再生できる）
- メモ入力（自由記述。転調情報・注意事項など）
- 保存／削除

#### 「今の状態を保存」ボタン

- 練習中画面（ピッチパイプ・メトロノーム）の下部固定コントローラーに常時表示
- タップすると現在鳴っているキー音・BPM・基準周波数をキャプチャして編集画面を開く
- 編集画面上でグループ選択（既存 or 新規作成）・曲名・メモを入力して保存

#### データスキーマ

```json
{
  "groups": [
    {
      "id": "g001",
      "name": "Harmony5",
      "songs": [
        {
          "id": "s001",
          "title": "Jupiter",
          "bpm": 88,
          "keys": ["A♯", "D", "F"],
          "baseFreq": 442.0,
          "notes": "サビで転調あり、ラスサビは半音上"
        }
      ]
    }
  ]
}
```

#### データ保存方式

- `localStorage`（キー：`acapella:songs`）に JSON で保存
- エクスポート（JSONダウンロード）／インポート（JSONアップロード）機能を提供
- グループメンバーとのデータ共有はJSONファイルの送付で対応

#### データアクセス層の設計方針

localStorageの読み書きは `storage.js` として1ファイルに集約し、アプリ全体から直接 `localStorage` を操作しない。将来的な暗号化対応・バックエンド移行時の修正箇所を最小化するため。

```javascript
// storage.js（例）
function getSongs() { ... }
function saveSongs(data) { ... }
function getSettings() { ... }
function saveSettings(data) { ... }
function getApiKey() { ... }
function saveApiKey(key) { ... }
```

### 2.6 設定機能 ※新規

曲ライブラリと同じく、練習中タブとは独立した別画面として提供する。

#### 設定項目

| カテゴリ | 項目 | 内容 |
|---|---|---|
| ピッチパイプ | 基準周波数デフォルト | アプリ起動時の初期値（420〜460Hz） |
| ピッチパイプ | 音色デフォルト | サイン波 / 三角波 / 矩形波 |
| AI・API | Gemini APIキー | ユーザー自身が取得したキーを入力・保存 |

#### APIキーの取り扱い方針（BYOK：Bring Your Own Key）

- 各ユーザーが自身のGemini APIキーを取得して登録する運用
- AIのAPI利用料金はユーザー自身が負担（アプリ側での一括負担なし）
- キーは`localStorage`（キー：`acapella:settings`）に保存。読み書きは `storage.js` の `getApiKey()` / `saveApiKey()` を経由する
- 画面上に下記の警告文を常時表示する：

> ⚠️ APIキーはこの端末にのみ保存されます。共有端末での使用はお控えください。

- HTTPS通信の徹底によりキーの通信経路を保護
- 設定画面にGemini APIキーの取得手順へのリンクと簡易ステップガイドを掲載し、ユーザーの導入障壁を下げる
- **APIキー保存時にテスト呼び出し（最小限のリクエスト）を実行し、「接続成功 ✓」または「キーが無効です」をその場でフィードバックする**
- **将来的なセキュリティ強化（Web Crypto APIによる暗号化等）は、不特定多数への公開タイミングで改めて検討する。それまでは現構成を維持。**

### 2.7 共通UI仕様

#### 画面構成

```
【メイン画面（練習中）】
タブバー：[ ≡ ] ／ ピッチパイプ ／ メトロノーム ／ リズム
          ↑
          タブバー左端のハンバーガーアイコン。
          タップでメニューが展開し、サブ画面へアクセスできる。

【サブ画面（ハンバーガーメニュー経由）】
📚 曲ライブラリ　…ライブ・本番前の音出し確認
⚙️ 設定　　　　　…初期設定・APIキー管理
```

#### ハンバーガーメニュー

- タブバーの**左端**に `≡` アイコンボタンを配置する
- タップすると**モーダルシート（画面下部からスライドイン）** が開く
  - iOS標準のUIパターンに沿っており、片手操作しやすい
  - ドロワー（横スライド）は採用しない
- メニュー項目（フェーズ1.5時点）：
  - `📚 曲ライブラリ`
  - `⚙️ 設定`
- タブバー右端は将来的な機能（通知・ユーザーアイコン等）のために空けておく

#### 採用理由

- 共通コントローラーのスペースを圧迫しない（44px以下のタップターゲット問題を回避）
- 練習中の誤タップによる意図しない画面遷移を防ぐ
- タブ＝プライマリナビゲーション、ハンバーガー＝セカンダリ・設定系、というiOS標準の慣習に沿う

#### 共通コントローラー（画面下部固定）

```
[ BPM表示 ＋ メトロノームボタン ]　[ ■ 全停止 ]
```

- `■ 全停止`：メトロノーム停止 ＋ ピッチパイプ全音停止
- 曲ライブラリ・設定へのアクセスはハンバーガーメニューに移管し、コントローラーには追加しない

#### その他

- **スマホファースト：** タップターゲット最小44px
- 解析・ログタブは フェーズ2実装まで「準備中」プレースホルダーを表示
  （現行のまま維持し、タブ構成の変更は不要）

## 3. 非機能要件

| 項目 | 方針 |
|---|---|
| データ保存 | サーバー不使用。`localStorage` にJSON保存 ＋ JSONエクスポートでバックアップ |
| DBランニングコスト | **ゼロ**（外部DBは使用しない） |
| オフライン対応 | **不要**（利用者に事前周知） |
| APIコスト | BYOKモデルのため**アプリ側のAPIコストはゼロ**。ユーザーが自身のGemini無料枠内で運用。アプリの利用者増加によるコスト増なし |
| セキュリティ | BYOKモデルのためAPIキーはフロント（localStorage）に平文保存。HTTPS徹底 ＋ 警告文表示で対応。不特定多数への公開タイミングで暗号化等の強化を改めて検討 |
| 収益モデル | 広告収益（アカペラ関連イベント・楽部枠）。広告データはCloudflare Workers + KV で管理し、アプリ起動時に取得・表示。広告内容の更新はサーバー側のみで完結 |
| 運営コスト | Cloudflare Workers / KV の無料枠で広告配信をまかなう。個人運営での持ち出しコストをゼロに抑える |

## 4. 技術方針

### フロントエンド

- HTML / CSS / Vanilla JS（またはシンプルなフレームワーク）
- Web Audio API（ピッチパイプ・メトロノーム音生成）
- Media Session API（バックグラウンド再生）
- PWA（Service Worker）：**UI/UX目的のみ**（ホーム画面追加・起動高速化）
- localStorage：曲ライブラリ・設定データの永続化（上限約5MB）
- `storage.js`：localStorageアクセスを一元管理するデータアクセス層。将来的にIndexedDBへ移行する場合もこのファイルの内部実装を差し替えるだけで対応可能
- **フェーズ2以降、文字起こし全文データが蓄積されると5MB上限に達する可能性あり。そのタイミングでIndexedDBへの移行を検討する**

### バックエンド / API

- Gemini API（文字起こし：gemini-2.0-flash、解析：gemini-2.0-flash）
- File API使用：30〜60分ごとに分割アップロードしてリクエスト失敗リスクを低減
- **APIキーはフロントから直接Gemini APIを呼び出すBYOK構成**（バックエンドプロキシは不要）
- **Cloudflare Workers + KV**：広告データの配信・管理のみに使用。APIキーのプロキシとしては使用しない

### 配布方針

| フェーズ | 配布方法 | コスト |
|---|---|---|
| クローズドβ（現在） | URL直接共有 | ゼロ |
| 公開展開 | Google Play ＋ PWAホーム画面追加（iOS） | ほぼゼロ |
| 将来検討 | AppStore申請（審査方針を見極めてから判断） | 年99ドルのみ |

### データフォーマット

- 曲ライブラリ：JSON（上記スキーマ）
- 解析結果：JSON（上記スキーマ）
- エクスポート：JSONダウンロード（将来的にPDF出力も検討）

## 6. 懸念点・リスクと対策

| # | 懸念点 | 対策 |
|---|---|---|
| 1 | Gemini APIプロンプトの精度 | 実音源でスパイク（技術検証）を行う。JSON出力スキーマを明示的に指示 |
| 2 | 3時間音源の1リクエスト制限 | 30〜60分ごとに分割アップロードするUIを設計 |
| 3 | Gemini無料枠の仕様変更 | BYOKモデルのためアプリ側への影響なし。ユーザーへ事前周知 |
| 4 | APIキーのlocalStorage平文保存 | BYOKモデルかつクローズドβ期間は許容範囲。HTTPS徹底 ＋ 警告文表示で対応。不特定多数公開タイミングで暗号化を改めて検討 |
| 5 | JSON＋音源の組み合わせミス | 誤組合せはデータ破損なし（シーク位置がズレるだけ）。UIで警告表示は検討余地あり |
| 6 | スマホでの操作性 | カードの高さ・タップ領域・フィルタUI等をスマホ実機で随時検証 |
| 7 | localStorageの消失リスク | JSONエクスポートを早期に実装しバックアップを促す。PWAホーム画面追加で意図しないクリアのリスクを低減 |
| 8 | 公開後のAPIキー誤用 | 設定画面にBYOK説明文・Gemini APIキー取得手順リンクを掲載 |
| 9 | 広告収益とAPIコストの相殺リスク | BYOKモデルによりAPIコストはゼロのまま維持。広告コストはCloudflare無料枠で吸収。個人運営での持ち出しを発生させない構造を堅持 |
| 10 | AppStore審査リスク | まずGoogle Play + iOS PWA配布で公開。AppStore申請はBYOK構成での審査通過可否を見極めてから判断 |
| 11 | localStorage容量上限（約5MB） | フェーズ2以降、文字起こし全文の蓄積により上限に達する可能性あり。`storage.js` の抽象化により、IndexedDBへの移行は内部実装の差し替えのみで対応可能 |
